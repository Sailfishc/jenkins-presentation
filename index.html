真实的需求

曾经参与一个学生创业团队，主要使用 Python 做 Web 开发。

  • A 新建了一个特性分支，写完了功能代码
  • A 忘记了执行单元测试，就将此分支合并到 master
  • A 将代码部署到生产服务器
  • 网站出现了 bug，严重影响了用户体验

解决方案：将测试/部署等 「容易遗漏的、重复性的、枯燥的」 过程，交给机器自动完成。

  • 减少风险，缩短 「开发-测试-交付」 的周期
  • 增强项目的可见性，增强团队对开发产品的信心

Jenkins 简介

市场占有率最高（70%）的持续集成工具。

  • 安装简单
  • 配置简单
  • 极其丰富的插件
  • 非常高的开发频率（GitHub 项目地址: https://github.com/jenkinsci/jenkins）

安装和启动

针对 Windows、Ubuntu、CentOS 等数10种操作系统都提供了安装包。

举例：CentOS 6.5

yum install jenkins
service jenkins start

使用 Docker 安装

docker run -d -p 8080:8080 -t zaiste/jenkins

启动后，Jenkins 会运行在 8080 端口，可通过浏览器访问。

自动化测试、代码质量检测、发布

Job: Jenkins 中的核心概念，表示一种抽象的任务。

根据任务需要，建立如下三个job:

  • tm_test: 执行测试、代码质量检测等
  • tm_staging_deploy: 将代码部署到 staging 服务器上
  • tm_production_deploy: 将代码部署到 production 服务器上

工作流程简介

  1. 使用 Git 作为代码管理工具
  2. 开发新功能时，需新建一个特性分支，开发完毕后合并到 master
  3. 每当 master 分支的代码变动时，触发 tm_test 任务
  4. tm_test 若运行通过，则自动触发 tm_staging_deploy 任务，否则发送邮件报警
  5. 人工登陆运行在 staging 服务器上的网站，检查是否达到预期效果
  6. 若达到预期效果，手动触发 tm_production_deploy 任务
  7. 人工登陆运行在 production 服务器上的网站，检查是否达到预期效果

tm_test / 准备测试环境

在 tm_test 中添加 Execute shell 项，输入：

if [ ! -d "venv" ]; then
  virtualenv -p /usr/bin/python2.7 venv
fi

. venv/bin/activate
pip install -r requirements.txt

tm_test / 测试

在 tm_test 中添加 Execute shell 项，输入：

nosetests --with-xunit --with-coverage --cover-package=tm && coverage xml
  • 使用 nose 测试 Python 程序
  • --with-xunit 告诉 nose 输出 JUnit 形式的测试报告
  • --with-coverage 表示同时计算测试覆盖率
  • --cover-package=tm 表示仅对指定的 package 执行测试覆盖率检测
  • coverage xml 表示输出 xml 格式的 coverage 报告

tm_test / 代码质量检测

使用 Pylint 检测 Python 代码质量：

pylint tm2 > pylint.xml || exit 0

使用 JSHint 检测 JavaScript 代码质量：

jshint --reporter=jslint $WORKSPACE/tm2/static/js/ > jslint.xml || exit 0

Jenkins 配置文件备份

Jenkins 的配置是一项较为费时的过程。

需要对 Jenkins 的配置文件（xml）进行备份，以便出现意外后恢复。

使用 ThinBackup 插件实现备份。

Jenkins 配置文件变更记录

A 对 Jenkins 配置进行了更新。

第二天 Jenkins 服务器出现错误，A 着手解决，但是他忘记了到底对配置做了哪些更改，找了很久都没有发现错误所在。

启发：如果能够知道 Jenkins 配置文件的变更历史，对于 debug 有非常大的帮助。

→ 使用 SCM Sync configuration plugin 插件记录 Jenkins 配置文件的变更历史。

原理：每次更新配置，都会发送一个 commit 提交到 Git 仓库。

小结

关注用户体验（对外），也关注工作体验（对内）。

关注 DevOps = 提升产品的交付速度和质量 + 提升程序员的工作体验。

插件清单

  • Git Plugin: 使用 Git 作为源代码管理
  • SSH plugin: 远程 ssh 登录 server 执行命令
  • Parameterized Trigger Plugin: 触发下游 job
  • Cobertura Plugin: 代码测试覆盖率报告
  • Task Scanner Plugin: 检测代码中出现的特殊标记（如 TODO 等）
  • Violations: 代码质量检测报告
  • ThinBackup: 用于备份 Jenkins
  • SCM Sync configuration plugin: 将 Jenkins 的配置变更同步到代码仓库中
